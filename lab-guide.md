# Azure Migrate – Step-by-Step Lab Guide

This guide walks you through the core workflow of Azure Migrate: creating a project, importing discovery data from an offline collector, reviewing and organizing your inventory, and producing assessments and reports.

---

## Prerequisites

Before you begin, verify the following:

- You have an **Azure subscription** where you can create resources.
- Your account has the **Azure Migrate Owner** role (or a role with higher permissions) on the subscription. This is required to create an Azure Migrate project. Without this role, project creation will fail.
  > **Note:** Starting November 2025, only users assigned the *Azure Migrate Owner* role or higher can create projects.
- You have downloaded the **`Synthetic_40_VMs.zip`** sample file provided with this lab. This file was generated by the Azure Migrate Collector and contains a snapshot of 40 discovered virtual machines.
  > **Important:** Importing this ZIP file into Azure Migrate can take **up to 45 minutes** to complete. Plan accordingly before starting Part 3.
- (Optional) A web browser with access to the [Azure portal](https://portal.azure.com).

---

## Part 1 – Create an Azure Migrate Project

A project is the central workspace where discovery data, assessments, and migration metadata are stored.

### Steps

1. Sign in to the [Azure portal](https://portal.azure.com).

2. In the top search bar, type **Azure Migrate** and select it under **Services**.

3. On the **Azure Migrate** overview page, select **Create project**.

4. In the **Create project** panel:
   - **Subscription** – Select your Azure subscription.
   - **Resource group** – Select an existing resource group or create a new one.
   - **Project name** – Enter a descriptive name (for example, `migrate-lab-project`).
   - **Geography** – Select the region that best represents where your data should reside. This does not restrict which Azure regions you assess as migration targets.

5. Select **Create** and wait for the project to deploy (usually a few minutes). The **Azure Migrate: Discovery and assessment** tool is added to the project automatically.

> **Reference:** [Quickstart: Create an Azure Migrate project using portal](https://learn.microsoft.com/azure/migrate/quickstart-create-project)

## Part 2 – Enable Preview Features

Some features in Azure Migrate are in preview and require a special feature-flag URL to access.

### Steps

1. After your project is created, navigate to it in the portal.

2. Replace the URL in your browser address bar with the following feature-flag URL:

   **[https://aka.ms/migrate/ameReport](https://aka.ms/migrate/ameReport)**

3. The URL will load the Azure portal with preview features enabled. Once the page loads, navigate back to the Azure Migrate project you created in Part 1 (use the top search bar to find **Azure Migrate**, then select your project). Preview features such as the new inventory experience and enhanced reporting will now be active for your session.

> **Tip:** Bookmark this URL or use it each time you open your Azure Migrate project to ensure preview features remain accessible.

## Part 3 – Import the Offline Assessment ZIP File

The Azure Migrate Collector is a lightweight tool that performs an **offline, one-time snapshot** of your environment. The collected data is exported as a ZIP file. In this lab you will use the pre-built ZIP file included in the repo.

### Download the ZIP file

1. Download the **`Synthetic_40_VMs.zip`** sample file that was provided with this lab.
2. Save it to a location on your local machine that is easy to find (for example, your **Downloads** folder or Desktop).

> **Note:** This ZIP file contains a pre-built offline snapshot of 40 virtual machines and was generated by the Azure Migrate Collector tool.

### Import the ZIP file into Azure Migrate

1. From your Azure Migrate project **Overview** page, select the **Start discovery** dropdown button. A menu will appear with three options:
   - **Using appliance** – Continuous discovery, ideal for detailed inventory and dependencies.
   - **Using collector** – Offline discovery, best for quick total cost benchmarking.
   - **Using import** – CSV or RVTools import for lift and shift cost benchmarking.

   Select **Using collector**.

2. The **Discover** page will open, showing three numbered stages:
   - **1. Download collector** – Skip this; you already have the ZIP file.
   - **2. Perform data collection** – Skip this; the sample data has already been collected.
   - **3. Import the ZIP file generated by collector** – This is the step you need.

3. Under **3. Import the ZIP file generated by collector**, select **Browse**, locate the downloaded **`Synthetic_40_VMs.zip`** file, and select it.

4. Select **Import** and monitor the **Import status** shown at the bottom of the page.

   > **Warning:** Importing this ZIP file can take **up to 45 minutes** to complete. Do not close the browser tab or navigate away while the import is in progress. Wait for the **Import status** to show a successful completion before proceeding to the next part of this lab.

### When to import additional data

If you need to add more inventory later (for example, from a separate environment or hypervisor type):

1. Navigate to **Explore inventory** > **All inventory**.
2. Select **Discover** > **Using collector**.
3. Follow steps 3–4 above.
   > Multiple ZIP files from different hypervisor types (VMware, physical) can be imported into the same project.

> **Reference:** [Discover servers using Azure Migrate collector](https://learn.microsoft.com/azure/migrate/how-to-discover-using-collector)

## Part 4 – Review the Action Center

The **Action Center** provides a centralized view of all issues surfaced during discovery, dependency visualization, and assessment. Reviewing it early helps you understand data quality and prioritize remediation before creating assessments.

### Access the Action Center

1. From the **Azure Migrate** project overview, locate the **Action Center** tile on the dashboard and select it.

   Alternatively, look for the notification/alert badge on the project navigation pane.

### What to look for

The Action Center surfaces issues across three stages:

| Stage | Common Issues |
|---|---|
| **Discovery** | Invalid credentials (error 10005), servers powered off or in an unreachable state, VMware Tools version anomalies |
| **Dependency Visualization** | Dependency data collection failures |
| **Assessment** | Insufficient performance data, unsupported OS types, configuration warnings |

#### Key points

- **Invalid credentials (error code 10005 / 60002):** The appliance or collector could not authenticate to one or more servers. Verify that the credentials provided have the required permissions.
- **Servers powered off (error 9004 / 9010):** Servers were discovered but were not running at collection time. Performance data will not be available. This is normal for non-critical or decommissioned servers.
- **VMTools anomalies:** VMware Tools on a guest VM is outdated or in an unexpected state. This can affect the accuracy of guest-level data collection.

> **Guideline:** Issues affecting more than **5–10% of your server population** should be investigated and remediated before proceeding. A small percentage of issues in a real environment is expected and generally acceptable.

### Sorting and filtering issues

- Use the **Severity**, **Affected workloads**, and **Timestamp** filters to prioritize high-severity problems.
- Switch to the **View by affected workloads** tab to see all issues grouped per server or workload.

> **Reference:** [Use Action Center for centralized issue tracking and resolution](https://learn.microsoft.com/azure/migrate/centralized-issue-tracking)

## Part 5 – Explore the Discovered Inventory

Once discovery data is imported, review what was collected across all workload types.

### All Inventory view

1. In your Azure Migrate project, navigate to **Explore inventory** > **All inventory**.

2. This view shows **everything** that was discovered: servers, database instances, web applications, and file shares – collectively referred to as **workloads**.

   | Inventory section | What it contains |
   |---|---|
   | **Infrastructure** | Servers (Windows, Linux, VMware VMs, physical) |
   | **Databases** | SQL Server, PostgreSQL, MySQL instances |
   | **Web apps** | .NET / IIS, Java web applications |
   | *(File shares)* | Discovered file shares |

3. The default columns shown include: **Workload**, **Category**, **Type**, **Edition**, **Version**, **Dependencies**, **Support Status**, **Discovery source**, and **Tags**.

4. Select **Columns** on the toolbar to add optional attributes (for example, CPU, memory, OS version) to the view.

### Drill down into a workload

- Select a workload name to open a detailed pane showing all discovered attributes, metadata, and any applied tags.

### Infrastructure (Servers) view

- Navigate to **Explore inventory** > **Infrastructure inventory** to see only servers.
- Key columns include: **Server**, **Operating system**, **IP address**, **DB instances** (running on that server), **Web apps** (running on that server), **Issues**, and **Support Status**.

### Databases view

- Navigate to **Explore inventory** > **Databases** to see all database instances discovered across your server estate.

### Web apps view

- Navigate to **Explore inventory** > **Web apps** to see all web application workloads discovered.

### Export inventory

- Select **Export data** to download a full CSV export of the inventory. This is useful for offline review, bulk tagging, and stakeholder reporting.

> **Reference:** [Review discovered inventory (preview) in Azure Migrate](https://learn.microsoft.com/azure/migrate/how-to-review-discovered-inventory)

## Part 6 – Tag Workloads

Tags allow you to organize and scope your inventory for assessments, business cases, and reporting. Tags can be applied manually in the portal or in bulk via CSV export/import.

### Why tag workloads?

- Scope assessments to a **subset** of discovered data (e.g., only production servers).
- Group workloads by department, application, environment, or migration wave.
- Use **special reserved tags** to convey migration intent and environment type to Azure Migrate.

### Reserved (special) tags

Azure Migrate recognizes two special tag keys that unlock additional planning capabilities:

| Tag key | Allowed values | Purpose |
|---|---|---|
| `AzM_Environment` | `Prod` or `Dev` | Marks the workload's environment |
| `AzM_MigrationIntent` | `Retain` or `Retire` | Marks the intended migration disposition |

> **Preview notice:** As of the writing of this lab, these reserved tags (`AzM_Environment` and `AzM_MigrationIntent`) are **still in preview**. They are only recognized and applied by Azure Migrate when you are using the feature-flag URL from Part 2. If you access the project via the standard portal URL, these tags will not influence inventory filtering, assessments, or business cases. Official Microsoft documentation does not yet reference these tag keys; they will be documented publicly once the feature reaches general availability.

---

### Option A – Apply tags manually in the portal

1. In **All inventory** (or any inventory view), select one or more workloads using the checkboxes.

2. Select **Tags** > **Add and edit tags** from the toolbar.

3. Enter the **key** and **value** for each tag. For the special tags, use the exact key names above.

4. Select **Save**.

To view tags for a specific workload, select its name and navigate to the **Tags** tab in the details pane.

To filter the inventory by tag:
1. Select **Add filter** next to the search bar.
2. Select the tag key.
3. Select the desired tag values.

---

### Option B – Apply tags in bulk via export/import

This is the recommended approach when tagging a large number of workloads.

1. In **All inventory**, select **Export data** to download the full inventory CSV.

2. Open the CSV file in Excel or another spreadsheet tool. The following columns are relevant for bulk tagging:

   | Column | Action |
   |---|---|
   | **Tags** | Add custom key/value pairs (e.g., `Department:Finance`) |
   | **Environment** | Set to `Prod` or `Dev` |
   | **Migration Intent** | Set to `Retain` or `Retire` |

3. Save the file after making your edits.

4. Return to the **All inventory** view in the portal.

5. Select **Tags** > **Import tags**.

6. Select **Browse**, choose your edited CSV file, and select **Import**.

   > **Note:** Application names in the CSV are case-sensitive. Avoid running multiple import operations in parallel.

> **Reference:** [Tags in Azure Migrate](https://learn.microsoft.com/azure/migrate/resource-tagging)

## Part 7 – Create Application Groupings

Application groupings let you logically associate related workloads (servers, databases, web apps) into a named application. This is useful for scoping assessments and business cases to a specific application.

### Access the Applications area

1. In your Azure Migrate project, navigate to **Explore applications** > **Applications**.

---

### Option A – Manually create an application (static grouping)

1. Select **Create application**.

2. Provide the following details:
   - **Application name** – A unique, descriptive name.
   - **Publisher** (optional) – For example, `SAP` for a packaged application or `Microsoft` for a custom .NET/SQL application.
   - **Technology stack** (optional) – For example, `.NET`, `SQL`, `MySQL`, `Tomcat`.

3. After filling in the properties, optionally add **tags** to the application.

4. In the workload selection step, search and select the servers, databases, and web apps that belong to this application.

5. Review the details and select **Create**.

---

### Option B – Create applications in bulk via export/import

1. In the **Applications** area, select **Import** > **Import applications**.

2. Select **Export all inventory** to download a CSV containing all discovered workloads.

3. In the exported CSV, fill in the **Application name** column for each workload row. A workload can belong to multiple applications – separate names with a comma (e.g., `App01, App02`).

4. Save the file and return to the import pane.

5. Select **Browse**, choose the edited CSV, and select **Import**.

6. After import completes, review the **number of applications created** and check the error file if any failures occurred.

> **Reference:** [Explore application inventory](https://learn.microsoft.com/azure/migrate/define-manage-applications)

## Part 8 – Create an Assessment

Assessments analyze your inventory and produce Azure readiness ratings, right-sized Azure targets, and estimated monthly costs.

### Steps

1. In the left navigation pane, select **Decide and plan** > **Assessments**.

2. On the Assessments page, select **+ Create assessment**.

3. The **Create assessment** wizard opens on the **Basics** tab. Provide the following:
   - **Assessment name** – Enter a friendly name (e.g., `Prod-Servers-AzureVM-March2026`).
   - **Add workloads** – Select **+ Add workloads** to choose the servers you want to include. Use filters or tags to scope the selection (for example, filter by `AzM_Environment = Prod`). You can also select **+ Add applications** to include application groupings.

4. Select **Next: General** to configure assessment properties:
   - **Target region** – The Azure region to size and price against.
   - **Default environment** – Production or Dev/Test (affects pricing).
   - **Sizing criteria** – *Performance-based* (uses collected performance data) or *As-is on-premises* (uses configured specs).
   - **Pricing options** – Azure offer type, currency, and applicable discounts.
   - **Azure Hybrid Benefit** – Apply if you have existing Windows Server or SQL Server licenses with Software Assurance.

5. Select **Next: Advanced** to review workload-specific settings for the assessment type (Azure VM, Azure SQL, etc.).

6. Select **Review + Create assessment**, review the summary, then select **Create assessment**.

7. Once created, the assessment will appear in the **Assessments** list under **Decide and plan** > **Assessments**.

> **Note:** After creating an assessment, it may take **several minutes or more** before processing is complete and results are ready to review. Refresh the Assessments list periodically and wait for the status to show as completed before moving on to Part 9.

> **Reference:** [Create an Azure VM assessment](https://learn.microsoft.com/azure/migrate/how-to-create-assessment) | [Best practices for creating assessments](https://learn.microsoft.com/azure/migrate/best-practices-assessment)

## Part 9 – Review Assessment Reports

After an assessment completes, review the results to understand migration paths, readiness, cost estimates, and emissions data.

### Access the assessment report

1. Navigate to **Decide and plan** > **Assessments**.

2. Select the assessment name to open the report.

### Overview page

The **Overview** page is the starting point for reviewing results. It contains several key areas:

#### Workloads summary
At the top left you will see the total number of workloads included in the assessment, along with a **View details** link to drill into the full workload list.

#### Migration preference
Use the **Migration preference** dropdown to switch the recommended path between options such as **Modernize (AI ready)**, **Lift and shift**, and others. The recommended path and cost estimates update dynamically based on your selection.

#### Migration paths
The assessment presents up to three side-by-side migration path cards:

| Path | Description |
|---|---|
| **PaaS preferred (Recommended)** | Maximum readiness coverage with preference to PaaS targets. Balances replatforming and rehosting. |
| **PaaS only** | All workloads targeted to PaaS services only. |
| **Lift and shift to Azure VM** | Quick IaaS migration of all workloads to Azure VMs. |

Each card shows:
- **Ready %** – Percentage of workloads assessed as ready for that path.
- **Cost** – Estimated total monthly cost (USD) for compute, storage, and security.
- **Migration readiness and strategy breakdown** – Split of workloads by strategy: Replatform, Refactor, Rehost, Retain, Retire.
- **Monthly cost estimate** – Itemized by Compute, Storage, and Security.
- **Monthly emissions estimate** – Carbon footprint in KgCO2e for compute and storage.
- **Performance coverage %** – How much of the sizing recommendation is based on collected performance data vs. defaults.

> **Tip:** A **Recommended path** star badge marks the path Azure Migrate considers most suitable given your migration preference. Select **View details** under any path card to explore workload-level results for that specific path.

#### Toolbar actions
Along the top of the assessment you will find:
- **Settings** – Modify assessment configuration.
- **Recalculate** – Re-run the assessment with the current settings.
- **Create wave** – Begin migration wave planning from the assessed workloads.
- **Generate IaC** – Generate Infrastructure as Code templates for the assessed workloads.
- **Export** – Download the assessment results as an Excel file for offline review or stakeholder sharing.
- **Refresh** – Reload the latest results.

### Drill into a migration path

1. Select **View details** on any path card to open the detailed workload-level view for that path.

2. Review individual workload readiness, recommended Azure target (VM size, PaaS service, etc.), and per-workload monthly cost.

3. Select any individual workload to see the full recommendation details including target SKU, disk configuration, and any issues or conditions affecting readiness.

> **Reference:** [Review Azure VM assessment](https://learn.microsoft.com/azure/migrate/review-assessment) | [Assess VMware VMs for migration to Azure VMs](https://learn.microsoft.com/azure/migrate/tutorial-assess-vmware-azure-vm)

## Part 10 – Create PPTX Reports from Assessments

Azure Migrate can generate polished PowerPoint (PPTX) presentation reports directly from your assessment data. These reports are ideal for sharing migration plans and cost summaries with stakeholders.

> **Preview notice:** As of the writing of this lab, the Reports feature is **not yet publicly released** and is only accessible via the feature-flag URL used in Part 2. It is not currently documented in the official Microsoft docs. This is another key reason the feature-flag URL is required throughout this lab.

### Steps

1. In the left navigation pane, scroll down to **Manage** and select **Reports (Preview)**.

2. The Reports page will open. If no reports have been created yet, you will see a **"No reports have been created yet"** placeholder with a **Generate Report** button.

3. Select **Generate Report** (either the button in the toolbar or the one in the center of the page).

4. In the report configuration panel that opens, fill in the following fields:

   - **Name** – Enter a descriptive name for the report (e.g., `Migration-Report-March2026`).
   - **Type** – Select the report types to include. By default both are checked:
     - **Azure Modernization and Migration Report**
     - **Security Insights Report**
     > **Note:** The **Azure VMware Solution (AVS) migration report** cannot be created alongside other report types. Leave it unchecked for this lab.
   - **Migration preference** – Select the migration preference that should drive the report (e.g., **Modernize (AI ready)**).
   - **Configuration** – Select **Generate report using scope and settings from an existing assessment**, then choose the assessment you created in Part 8 from the dropdown.

5. Select **Generate Report**.

6. The report will appear in the **Reports (Preview)** list. Report generation may take **several minutes** to complete. Refresh the page periodically and wait for the status to indicate the report is ready before attempting to download it.

7. Once complete, select the report name to download the generated PPTX file.
